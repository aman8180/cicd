What is CI/CD? (continuous integration and deployment)

Published on November 2, 2025

If you've spent any time in the software development world, you've almost certainly heard the term "CI/CD." It stands for Continuous Integration and Continuous Delivery (or Continuous Deployment). These practices are the backbone of modern DevOps, designed to help development teams deliver code changes more frequently and reliably.

What is Continuous Integration (CI)?

Continuous Integration is the practice of automating the integration of code changes from multiple contributors into a single software project. Here's how it works:

  1. A developer commits code to a shared repository (like Git).
  2. A CI server (like Jenkins, GitLab CI, or GitHub Actions) automatically detects the change.
  3. The server builds the code and runs a suite of automated tests (unit tests, integration tests, etc.).
  4. If the build or tests fail, the team is notified immediately to fix the issue.

The primary goal of CI is to prevent integration problems by finding and addressing bugs quicker, improving software quality, and reducing the time it takes to validate and release new updates.

What is Continuous Delivery (CD)?

Continuous Delivery picks up where Continuous Integration leaves off. It's the practice of automatically deploying every code change that passes the CI stage to a testing or "staging" environment. This ensures that you always have a deployable build that has passed all automated tests.

The final step—deploying to the live production environment—is still a manual button-push. This gives the team control over *when* the new code goes live.

What about Continuous Deployment (the *other* CD)?

Continuous Deployment goes one step further than Continuous Delivery. With this practice, every change that passes all stages of your automated pipeline is *automatically* deployed to production. There is no manual intervention.

This approach is excellent for mature teams who have a high degree of confidence in their automated testing pipeline.

A Simple CI/CD Pipeline Example (YAML)

Here is a very simple example of what a CI pipeline might look like in a `.gitlab-ci.yml` file: 


# This pipeline has two stages: build and test
stages:
  - build
  - test

build_job:
  stage: build
  script:
    - echo "Building the application..."
    - npm install
    - npm run build

test_job:
  stage: test
  script:
    - echo "Running tests..."
    - npm test

Why Does CI/CD Matter?

Ultimately, a CI/CD pipeline automates the steps in your software delivery process. This automation helps you:

  • Release Faster: Automate builds and tests to get new features to users quickly.
  • Improve Quality: Catch bugs early in the development cycle before they become big problems.
  • Reduce Risk: Smaller, more frequent code changes are easier to troubleshoot than large, infrequent releases.
  • Boost Productivity: Developers can focus on writing code instead of worrying about manual deployment logistics.
